# 上位机 - ROS - 控制模块

## 1. 这个功能到底解决什么问题

这一部分负责把 ROS 世界里的高层速度指令，变成底盘能真正执行的电机控制量，同时把 MCU 回传的编码器和状态数据重新转换成里程计与 TF。它是“感知/导航/Agent 决策”和“电机实际运动”之间的桥。如果这里不稳定，上层所有智能模块都只能停留在仿真层面。

## 2. 先看哪些文件

- `src/base_control/src/base_control.cpp`
- `src/base_control/src/pid.cpp`
- `src/base_control/src/uart.cpp`
- `src/base_control/cfg/Base.cfg`
- `src/base_control/launch/base_control.launch`

## 3. 启动路径 / 数据流 / 控制流

控制主链路是：`/cmd_vel` -> 目标线速度 / 角速度 -> 左右轮目标脉冲 -> PID 计算 PWM -> UART 下发给 MCU -> MCU 返回编码器脉冲 -> 上位机积分得到里程计和 TF。

典型启动方式：

```bash
catkin_make
source devel/setup.bash
roslaunch base_control base_control.launch
```

如果要联调整机，通常它会作为 `pkg_launch` 或导航系统的底盘控制基础服务存在。

## 4. 亮点实现细节

### 4.1 从速度到底盘脉冲的转换是显式建模的

`base_control.cpp` 不是把 `/cmd_vel` 直接粗暴映射成两个轮子的 PWM，而是先依据 `pluses_m`、`wheel_distance_m` 等参数，把线速度和角速度转换为左右轮目标脉冲。也就是说，这里先做了一个简化差速底盘运动学映射，再把结果交给 PID。

### 4.2 PID 控制和运动学被清晰拆开

`pid.cpp` 只关心控制律，`base_control.cpp` 负责运动学和系统拼装。这种分层很值得学，因为它让你可以单独调 PID，也可以单独校准轮径、脉冲数和轴距，而不会把所有误差混在一块。

### 4.3 方向切换时会重置 PID 状态

这是一个很容易被忽略、但非常工程化的细节。轮子从前进切换到后退时，如果还沿用之前的积分和误差状态，控制器往往会出现冲击和振荡。这里在方向反转时重置 PID，有利于降低切换瞬态问题。

### 4.4 UART 协议把上位机和 MCU 解耦

`uart.cpp` 负责打包 `CmdData`、解析 `McuData`。这样高层 ROS 节点不用关心串口字节流细节，只需要处理“期望控制量”和“回传状态”。这是一种很典型的机器人分层接口设计。

### 4.5 里程计和 TF 是由真实编码器反馈驱动的

编码器不是只用于控制闭环，也被用于里程计积分和 TF 发布。也就是说，导航、定位、跟踪里依赖的底盘运动信息，并不是拍脑袋估计，而是来自底层实际反馈。

### 4.6 动态参数可在线调

`Base.cfg` 提供轮周长、脉冲数、PID 参数、最大 PWM 等动态参数。对调车来说，这非常关键，因为底盘误差很多来自机械装配和地面条件，在线调参比反复改代码重编译高效得多。

### 4.7 x86 回退路径便于桌面联调

当串口不存在时，x86 路径会注入零值或回退行为，使上层系统在开发机上仍可启动。这种设计虽然不显眼，但对大型项目非常重要，它让感知、导航、Web、Agent 能在没有真实底盘的环境下继续开发。

## 5. 调试与验证方法

- 单独给 `/cmd_vel` 发送固定线速度和角速度，检查左右轮目标脉冲是否符合预期。
- 调 PID 时先只测单方向直行，再测原地旋转，最后再测速度切换。
- 检查串口收发是否稳定，避免把电机问题误判成控制算法问题。
- 比对里程计和实际位移，优先校准轮径、脉冲数、轮距，再调 PID。
- 桌面联调时要明确知道自己走的是 x86 回退路径，不要把“能启动”误当成“真控制链路可用”。

## 6. 学习路线（30 分钟 / 2 小时 / 1 天）

### 30 分钟

先读 `base_control.cpp`，搞清楚 `/cmd_vel` 如何映射到左右轮目标值。

### 2 小时

补读 `pid.cpp`、`uart.cpp` 和 `Base.cfg`，把控制律、串口协议、参数系统三者串起来。到这一步，你应该能说清楚这个节点如何同时承担控制和状态回传。

### 1 天

亲自做一次标定和调参：测轮径误差、测脉冲数、调 PID，并观察方向切换和急停时的表现。只有做过这一步，你才会真正理解这部分为什么是整机稳定性的关键。

## 7. 你学完后应该能回答什么问题

- `/cmd_vel` 为什么不能直接映射成 PWM。
- 运动学映射和 PID 为什么必须分层。
- 方向切换时为什么要重置 PID 状态。
- 里程计为什么必须基于编码器反馈而不是命令速度。
- 为什么一个好的上位机控制节点必须同时考虑真实底盘和桌面联调两种运行环境。

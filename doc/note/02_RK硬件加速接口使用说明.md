# 项目中的 RK 硬件加速接口使用说明

这份文档是给小白看的，目标不是只告诉你“用了 RK”，而是把当前项目里和 Rockchip 硬件加速有关的代码、接口、调用顺序、参数、常见坑，一次讲清楚。

本文以当前主链路的 ROS1 工作区 `src/` 为主，因为真正启动的是 [all.launch](../../src/pkg_launch/launch/all.launch#L1)。ROS2 工作区 `ros2_ws/src/img_decode` 里也有同名的 `mpp_decode` / `rga_resize` 镜像实现，但核心思路和接口基本一致。

## 1. 先给你结论：项目里到底用了哪些 RK 硬件单元

当前项目真正和 RK 硬件加速直接相关的，主要是 4 类能力：

| 模块 | 硬件/SDK | 作用 | 代码位置 |
| --- | --- | --- | --- |
| `img_decode` | MPP | 把 MJPEG/JPEG 压缩帧硬解码成 RGB | [mpp_decode.h](../../src/img_decode/src/mpp_decode.h), [mpp_decode.cpp](../../src/img_decode/src/mpp_decode.cpp) |
| `img_decode` | RGA | 把解码后的 RGB 大图硬缩放成小图 | [rga_resize.h](../../src/img_decode/src/rga_resize.h), [rga_resize.cpp](../../src/img_decode/src/rga_resize.cpp) |
| `rknn_yolov6` | RKNN + RGA | 用 NPU 推理 YOLOv6；必要时先用 RGA 把图缩到模型输入尺寸 | [rknn_run.h](../../src/rknn_yolov6/src/rknn_run.h), [utils.h](../../src/rknn_yolov6/src/utils.h), [utils.cpp](../../src/rknn_yolov6/src/utils.cpp) |
| `img_encode` | RGA + MPP | 先把 RGB 转成 YUV420P，再硬编码成 JPEG | [rga_cvtcolor.h](../../src/img_encode/src/rga_cvtcolor.h), [mpp_encode.h](../../src/img_encode/src/mpp_encode.h) |

你可以把这 4 步先记成一句话：

```text
MPP 解码 -> RGA 缩放 -> RKNN 推理 -> RGA 转色 + MPP 编码
```

## 2. 这条链路里，哪些模块是 RK 加速，哪些不是

当前主链路见 [all.launch](../../src/pkg_launch/launch/all.launch#L6)：

```text
usb_camera
  -> /image_raw/compressed
img_decode
  -> /camera/image_raw
rknn_yolov6
  -> /camera/image_det + /ai_msg_det
object_track
  -> /camera/image_det_track
img_encode
  -> /camera/image_det_track/compressed
```

其中：

- `usb_camera` 负责采集 MJPEG，相机采集本身不是 RK 硬件加速。
- `img_decode` 是第一个 RK 加速模块，负责 `MPP 解码 + RGA 缩放`。
- `rknn_yolov6` 是第二个 RK 加速模块，负责 `RGA 预处理 + RKNN NPU 推理`。
- `object_track` 不属于 RK 硬件加速模块，它做的是图像/检测框/激光雷达融合。
- `img_encode` 是第三个 RK 加速模块，负责 `RGA 转色 + MPP JPEG 编码`。

## 3. 编译开关和依赖：什么时候才会真正走 RK 硬件

三个核心包都用了同一个编译开关：`USE_ARM_LIB`。

### 3.1 `img_decode`

见 [img_decode/CMakeLists.txt](../../src/img_decode/CMakeLists.txt#L44)：

- 如果 `CMAKE_HOST_SYSTEM_PROCESSOR` 匹配 `aarch64`，就定义 `USE_ARM_LIB=1`
- 并链接：
  - `rga`
  - `rockchip_mpp`
- 否则定义 `USE_ARM_LIB=0`，退化为 OpenCV 软解码 / 软缩放

### 3.2 `rknn_yolov6`

见 [rknn_yolov6/CMakeLists.txt](../../src/rknn_yolov6/CMakeLists.txt#L37)：

- `aarch64` 时定义 `USE_ARM_LIB=1`
- 并链接：
  - `rga`
  - `rknn_api`
  - `rknnrt`
- 否则用 `USE_ARM_LIB=0`

### 3.3 `img_encode`

见 [img_encode/CMakeLists.txt](../../src/img_encode/CMakeLists.txt#L26)：

- `aarch64` 时定义 `USE_ARM_LIB=1`
- 并链接：
  - `rga`
  - `rknn_api`
  - `rknnrt`
  - `rockchip_mpp`
- 否则退化为 OpenCV 软编码

### 3.4 你要记住的工程事实

- 只有 `USE_ARM_LIB==1` 时，项目才会编译真正的 Rockchip 加速代码。
- 在 PC/x86 上阅读源码时，很多 RK 接口只是“能看到声明”，但不会真的跑到那条分支。
- 项目里“是否走硬件”的切换，不是运行时参数，而是编译时决定。

## 4. 先按层次理解接口，不容易乱

项目里的“接口”可以分成三层：

### 4.1 第 1 层：节点封装接口

这是你在项目里最直接看到的类和函数，例如：

- `ImgDecode`
- `RknnRun`
- `ImgEncode`

它们负责把 ROS 消息、线程、队列、参数管理和硬件接口串起来。

### 4.2 第 2 层：项目自己封装的 RK 工具接口

这是作者为了让业务代码更好用，额外包了一层：

- `MppDecode`
- `MppEncode`
- `rga_resize(...)`
- `rga_cvtcolor(...)`
- `rknn_load(...)`
- `rknn_config(...)`
- `post_process(...)`

这层是你最值得认真学的，因为它正好是“业务代码”和“底层 SDK”之间的桥。

### 4.3 第 3 层：Rockchip 官方 SDK 接口

例如：

- MPP 的 `mpp_create`、`mpp_init`、`mpp_buffer_get`
- RGA 的 `wrapbuffer_virtualaddr`、`imresize`、`imcopy`、`imcvtcolor`
- RKNN 的 `rknn_init`、`rknn_query`、`rknn_inputs_set`、`rknn_run`

这一层最底层，也最容易把人看懵。建议你先看第 1 层，再看第 2 层，最后回头对照第 3 层。

### 5.1 `ImgDecode` 类接口

定义见 [img_decode.h](../../src/img_decode/src/img_decode.h#L19)。

#### 公开接口

| 接口 | 作用 | 你要怎么理解 |
| --- | --- | --- |
| `ImgDecode()` | 读取 launch 参数，创建发布器，初始化 `MppDecode` | 节点启动入口 |
| `void compressed_image_callback(const sensor_msgs::CompressedImageConstPtr& msg)` | 处理一帧压缩图像 | 真正的数据处理入口 |
| `void run_check_thread()` | 根据下游订阅者数量，决定是否继续订阅上游压缩流 | 节省 CPU、MPP、RGA 资源 |

#### 关键成员

| 成员 | 含义 |
| --- | --- |
| `compressed_image_sub` | 上游压缩图订阅者 |
| `raw_image_pub` | 解码缩放后的 RGB 图发布器 |
| `msg_pub` | 复用的 `sensor_msgs::Image` 消息缓存 |
| `image` | 当前解码得到的 `cv::Mat` |
| `fps_div` | 跳帧分频系数 |
| `scale` | 缩放比例 |
| `mpp_decode` | Rockchip MPP 解码器对象 |

#### 典型调用顺序

1. 构造函数里读取参数：
   - `sub_jpeg_image_topic`
   - `pub_raw_image_topic`
   - `fps_div`
   - `scale`
   - `width`
   - `height`
2. `USE_ARM_LIB==1` 时执行 `mpp_decode.init(width, height)`
3. 回调里先分频
4. 用 `mpp_decode.decode(...)` 把 JPEG 解成 RGB
5. 用 `rga_resize(...)` 把 RGB 大图直接缩到消息输出缓冲区
6. 发布 `/camera/image_raw`

#### 对应 launch 参数

见 [img_decode.launch](../../src/img_decode/launch/img_decode.launch#L3)：

| 参数 | 默认值 | 作用 |
| --- | --- | --- |
| `sub_jpeg_image_topic` | `/image_raw/compressed` | 上游压缩流话题 |
| `pub_raw_image_topic` | `/camera/image_raw` | 输出 raw RGB 图话题 |
| `fps_div` | `1` | 跳帧，`2` 表示每 2 帧处理 1 帧 |
| `scale` | `0.5` | 缩小到一半，`1280x720 -> 640x360` |
| `width` | `1280` | 输入图像宽 |
| `height` | `720` | 输入图像高 |

### 5.2 `MppDecode` 类接口

定义见 [mpp_decode.h](../../src/img_decode/src/mpp_decode.h#L49)。

这个类是项目对 MPP 解码器的封装。你可以把它理解为：

```text
JPEG 字节流 -> MPP -> RGB888 cv::Mat
```

#### 公开接口

| 接口 | 参数 | 返回值 | 说明 |
| --- | --- | --- | --- |
| `MppDecode()` | 无 | 无 | 构造函数，本身不做重活 |
| `~MppDecode()` | 无 | 无 | 释放 packet、frame、ctx、buffer group |
| `void init(int width, int height)` | 输入图像宽高 | 无 | 初始化整条解码流水线 |
| `int decode(unsigned char* srcFrm, size_t srcLen, cv::Mat& image)` | JPEG 数据地址、长度、输出图像 | `0` 成功，负值失败 | 解码一帧 JPEG |

#### 私有接口

| 接口 | 作用 |
| --- | --- |
| `int init_mpp()` | 创建 MPP 上下文，并把输出格式配置成 RGB888 |
| `int init_packet_and_frame(int width, int height)` | 按 stride 对齐申请输入输出缓冲区 |
| `int get_image(MppFrame& frame, cv::Mat& image)` | 把 MPP 输出的 frame 包装成 `cv::Mat` |

#### `MppDecode::init(width, height)` 做了什么

内部顺序是：

1. `init_mpp()`
2. `init_packet_and_frame(width, height)`

也就是先创建上下文，再创建输入输出缓冲区。

#### `MppDecode::decode(...)` 该怎么理解

这个函数不是“给一张图就直接返回一张图”那么简单，它内部走了完整的 MPP 任务队列：

1. 先把 JPEG 数据 `memcpy` 到 `pktBuf`
2. `mpp_packet_set_pos / mpp_packet_set_length`
3. `mpi->poll(ctx, MPP_PORT_INPUT, MPP_POLL_BLOCK)`
4. `mpi->dequeue(ctx, MPP_PORT_INPUT, &task)`
5. `mpp_task_meta_set_packet(task, KEY_INPUT_PACKET, packet)`
6. `mpp_task_meta_set_frame(task, KEY_OUTPUT_FRAME, frame)`
7. `mpi->enqueue(ctx, MPP_PORT_INPUT, task)`
8. 再等输出端：
   - `poll`
   - `dequeue`
9. 最后 `get_image(...)` 把输出包装成 `cv::Mat`
10. 把 `task` 再放回输出队列

#### 为什么 `get_image()` 只包成 `cv::Mat`，不立刻 `memcpy`

因为作者已经在注释里点明了一个关键点：

- MPP 输出 buffer 往往不是 cache friendly 的普通用户态内存
- 如果你马上 `memcpy` 到 CPU buffer，可能很慢
- 所以后面最好直接让 RGA 接着处理这块内存

这也是为什么 `img_decode` 里会紧接着调用 `rga_resize(...)`。

#### `MppDecode` 的一个最小理解例子

```cpp
MppDecode dec;
dec.init(1280, 720);

cv::Mat rgb;
int ret = dec.decode(jpeg_ptr, jpeg_len, rgb);
if (ret == 0) {
  // rgb 现在引用的是 MPP 输出 buffer
}
```

### 5.4 `rga_resize(cv::Mat&, unsigned char*, float)` 接口

定义见 [rga_resize.h](../../src/img_decode/src/rga_resize.h#L5)。

#### 接口原型

```cpp
int rga_resize(cv::Mat &img_in, unsigned char *img_out_data, float scale);
```

#### 参数说明

| 参数 | 含义 |
| --- | --- |
| `img_in` | 输入 RGB 图像，通常就是 `MppDecode` 输出的 `cv::Mat` |
| `img_out_data` | 输出缓冲区首地址，通常直接传 `sensor_msgs::Image::data.data()` |
| `scale` | 缩放比例，`1.0` 表示只拷贝，`0.5` 表示长宽各缩半 |

#### 返回值

- `0` 或 `IM_STATUS_NOERROR`：成功
- 其它值：RGA 出错

#### 它内部实际调用了哪些 RGA 官方接口

| 官方接口 | 作用 | 代码位置 |
| --- | --- | --- |
| `wrapbuffer_virtualaddr(...)` | 把输入输出地址包装成 RGA 可识别的 buffer | [rga_resize.cpp](../../src/img_decode/src/rga_resize.cpp#L32) |
| `imcheck(...)` | 检查输入输出格式和尺寸是否合法 | [rga_resize.cpp](../../src/img_decode/src/rga_resize.cpp#L43) |
| `imresize(src, dst)` | 真正执行硬缩放 | [rga_resize.cpp](../../src/img_decode/src/rga_resize.cpp#L53) |
| `imcopy(src, dst)` | 当 `scale == 1.0` 时走硬件拷贝 | [rga_resize.cpp](../../src/img_decode/src/rga_resize.cpp#L56) |

#### 最小理解例子

```cpp
cv::Mat rgb_big = ...;
std::vector<unsigned char> out(rgb_big.cols * 0.5 * rgb_big.rows * 0.5 * 3);
int ret = rga_resize(rgb_big, out.data(), 0.5);
```

### 6.1 `RknnRun` 类接口

定义见 [rknn_run.h](../../src/rknn_yolov6/src/rknn_run.h#L75)。

#### 公开接口

| 接口 | 作用 |
| --- | --- |
| `RknnRun()` | 读取模型、话题、阈值等参数，创建发布器 |
| `void sub_image_callback(const sensor_msgs::ImageConstPtr& msg)` | 只负责把图像入队 |
| `int run_infer_thread()` | 推理线程：取图、RGA 缩放、喂 RKNN、取输出 |
| `int run_process_thread()` | 后处理线程：解码框、画框、发布、释放输出 |
| `void run_check_thread()` | 没人看结果时，不继续订阅上游输入 |

#### 核心成员

| 成员 | 含义 |
| --- | --- |
| `ctx` | RKNN 上下文句柄 |
| `io_num` | 模型输入输出 tensor 个数 |
| `out_scales` / `out_zps` | 量化模型反量化参数 |
| `capture_data_queue` | 输入图像队列 |
| `process_data_queue` | 推理结果队列 |
| `image_pub` | 带框图像发布器 |
| `det_pub` | 结构化检测框发布器 |

#### `InferData` 结构体

定义见 [rknn_run.h](../../src/rknn_yolov6/src/rknn_run.h#L37)。

它是项目自己定义的“一帧推理上下文”：

| 字段 | 作用 |
| --- | --- |
| `outputs[5]` | RKNN 输出 tensor 缓冲 |
| `orig_img` | 原始 RGB 图 |
| `mod_size` | 模型输入尺寸 |
| `img_size` | 原图尺寸 |
| `header` | 原时间戳和 `frame_id` |

这里你一定要注意一个坑：

- `outputs[5]` 是固定数组
- 注释里已经明确说了：如果模型真实输出数大于 5，会越界崩溃
- 所以它不是通用设计，而是当前模型下的经验值

### 6.2 `rknn_yolov6` 节点的线程入口

主程序见 [main.cpp](../../src/rknn_yolov6/src/main.cpp#L6)。

它启动了三个线程：

1. `infer_thread`
2. `process_thread`
3. `check_thread`

这样做的目的，是把“推理”和“后处理/发布”拆开，避免所有工作堵在一个 ROS 回调线程里。

### 6.3 `rknn_load(...)` 接口

定义见 [utils.h](../../src/rknn_yolov6/src/utils.h#L22)，实现见 [utils.cpp](../../src/rknn_yolov6/src/utils.cpp#L5)。

#### 接口原型

```cpp
int rknn_load(rknn_context &ctx,
              const char* model_name,
              unsigned char* model_data,
              rknn_input_output_num &io_num,
              bool print_perf_detail,
              bool use_multi_npu_core);
```

#### 它做了什么

1. `load_model(...)` 把 `.rknn` 模型文件读进内存
2. 调用 `rknn_init(...)` 创建 RKNN 上下文
3. 调用 `rknn_set_core_mask(...)` 指定 NPU 核策略
4. 调用 `rknn_query(... RKNN_QUERY_SDK_VERSION ...)`
5. 调用 `rknn_query(... RKNN_QUERY_IN_OUT_NUM ...)`

#### 参数说明

| 参数 | 作用 |
| --- | --- |
| `ctx` | 输出参数，返回 RKNN 上下文 |
| `model_name` | `.rknn` 模型文件路径 |
| `model_data` | 模型内存指针，当前实现没有真正从外面传进来 |
| `io_num` | 输出参数，返回输入输出 tensor 数量 |
| `print_perf_detail` | 是否打开性能统计 |
| `use_multi_npu_core` | 是否启用多 NPU 核 |

#### 返回值

- `0`：成功
- `< 0`：失败

#### 这里用到的 RKNN 官方接口

| 官方接口 | 作用 | 代码位置 |
| --- | --- | --- |

### 6.4 `rknn_config(...)` 接口

定义见 [utils.h](../../src/rknn_yolov6/src/utils.h#L25)，实现见 [utils.cpp](../../src/rknn_yolov6/src/utils.cpp#L71)。

#### 接口原型

```cpp
int rknn_config(rknn_context &ctx,
                rknn_input_output_num &io_num,
                int &width,
                int &height,
                rknn_input *inputs,
                rknn_output *outputs,
                std::vector<float> &out_scales,
```

#### 它做了什么

1. 查询每个输入 tensor 的属性
2. 查询每个输出 tensor 的属性
3. 推断模型输入宽高和通道数
4. 初始化 `rknn_input` 数组
5. 初始化 `rknn_output` 数组

#### 你应该重点理解的几个结构体

#### `output_want_float` 有什么影响

- `0`：输出保持量化格式，通常是 `int8`
- `1`：驱动帮你转成 `float`

当前项目默认在 launch 里是 `true`，见 [rknn_yolov6.launch](../../src/rknn_yolov6/launch/rknn_yolov6.launch#L18)。

这会让后处理更容易理解，但也可能增加一点额外开销。

#### `MppEncode::init(...)` 做了什么

内部主要流程：

1. 计算 stride 和 `frame_size`
2. `mpp_buffer_get(...)` 申请输入 frame buffer
3. `mpp_create(...)`
4. `mpp_init(ctx, MPP_CTX_ENC, MPP_VIDEO_CodingMJPEG)`
5. 设置 `rc_cfg`
6. 设置 `prep_cfg`
7. 设置 `codec_cfg`
8. 设置 `sei_mode`
9. 初始化 `MppFrame`
10. 绑定输入 buffer

#### `MppEncode::encode(...)` 做了什么

1. 把输入 YUV420P 数据 `memcpy` 到 `buf_ptr`
2. `encode_put_frame(...)`
3. `encode_get_packet(...)`
4. `mpp_packet_get_pos(...)`
5. `mpp_packet_get_length(...)`
6. 拷贝到 `jpeg_data`
7. `mpp_packet_deinit(...)`

## 8. 一帧图像经过 RK 硬件时，完整调用顺序是什么

如果只看 RK 相关模块，一帧图像的大致顺序是：

```text
1. img_decode::compressed_image_callback(msg)
2. MppDecode::decode(jpeg_ptr, jpeg_len, image)
3. rga_resize(image, msg_pub.data.data(), scale)
4. 发布 /camera/image_raw

5. RknnRun::sub_image_callback(msg)
6. RknnRun::run_infer_thread()
7. rga_resize(orig_img, img_resize, model_size)    // 尺寸不同时
8. rknn_inputs_set(...)
9. rknn_run(...)
10. rknn_outputs_get(...)
11. process_data_queue.push(infer_data)

12. RknnRun::run_process_thread()
13. post_process(...)
14. rknn_outputs_release(...)
15. 发布 /camera/image_det 和 /ai_msg_det

16. ImgEncode::image_callback(msg)
17. rga_cvtcolor(rgb, yuv420p)
18. MppEncode::encode(yuv420p, size, jpeg_data)
19. 发布 /camera/image_det_track/compressed
```

这条顺序里，真正进 RK 硬件单元的时刻分别是：

- 第 2 步：进 MPP 解码器
- 第 3 步：进 RGA 缩放器
- 第 8 到 10 步：进 RKNN/NPU
- 第 17 步：进 RGA 做颜色转换
- 第 18 步：进 MPP 编码器

## 10. 新手最容易踩的坑

这部分很重要，因为你看懂接口，不等于你就能正确用。

### 10.1 `USE_ARM_LIB` 不是运行时开关

- 它是编译时开关
- 不是 launch 参数
- 你在 x86 上就算把 launch 改了，也不会变成 RK 硬件加速

### 10.2 `MppDecode` 输出的 `cv::Mat` 不等于普通可随便 `memcpy` 的图

作者已经在 [mpp_decode.cpp](../../src/img_decode/src/mpp_decode.cpp#L339) 注释里说了：

- 这块 buffer 更适合继续交给 RGA
- 不建议立刻大规模 `memcpy`

### 10.3 `rknn_outputs_get(...)` 后必须 `rknn_outputs_release(...)`

这不是可选项。

漏掉这一步，NPU 输出 buffer 会积压。

### 10.4 `InferData.outputs[5]` 是固定长度，模型输出变多会崩

这是当前实现里的一个硬编码风险点，不是 RKNN API 本身的问题。

### 10.5 `output_want_float=true` 方便理解，但不一定最省资源

- 方便后处理
- 但驱动要帮你做类型转换
- 如果你追极限性能，通常要重新评估是否直接处理量化输出

### 10.6 `MppEncode` 吃的是 `YUV420P`，不是 `RGB`

所以编码前必须先走 `rga_cvtcolor(...)`。

### 10.7 stride 对齐不能忽略

在 MPP/RGA 里，很多 buffer 申请不是简单 `width * height * channels`，而要考虑：

- `hor_stride`
- `ver_stride`
- `MPP_ALIGN(width, 16)`

### 10.8 当前实现里有一个值得注意的资源释放细节

在 [utils.cpp](../../src/rknn_yolov6/src/utils.cpp#L5) 里，`rknn_load(...)` 内部调用了 `load_model(...)`，但 `model_data` 参数是按值传入，不会回写给调用者。

这意味着：

- 当前代码更像是“把模型读进来后直接喂给 `rknn_init`”
- 但没有在同一层显式 `free` 这块模型内存

这不影响你理解接口本身，但从工程上看，这是一个可以继续优化的点。

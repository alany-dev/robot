**面试版**
如果我是候选人，在面试里我会这样讲：

这套低时延相机视觉链路，我不是只优化某一个算法节点，而是把整条链路都按“少传大数据、少排队、尽量走硬件、优先最新帧”来设计。主链路是：相机采集 `usb_camera`，解码缩放 `img_decode`，目标检测 `rknn_yolov6`，目标跟踪与控制 `object_track`，最后如果需要给上位机看图，再走 `img_encode` 做回传。默认启动顺序在 [all.launch](../../src/pkg_launch/launch/all.launch#L6)。

第一步，相机端我直接走 V4L2，从 `/dev/video0` 采 1280x720、30fps 的 MJPEG，而不是先采原始 RGB/YUYV。这样做的好处是，USB 侧和 ROS 第一跳传的都是压缩流，带宽压力小很多。驱动层用 `mmap` 方式申请环形 buffer，不是每帧都重新 malloc 和 read，所以采集端本身比较轻。采到的帧直接打时间戳，发布到 `/image_raw/compressed`。另外这个节点有一个很实用的策略，就是如果下游没人订阅，它会停采，不白白占相机和 CPU。

第二步，进入 `img_decode` 以后，我不在大分辨率上继续往后跑，而是立刻做“硬解码 + 硬缩放”。在 Rockchip 平台上，JPEG 解码走 MPP，缩放走 RGA，直接把 1280x720 缩到 640x360，再发布成 RGB 图像给推理节点。这样等于在链路前半段就把后面要处理的像素量砍成四分之一，推理、跟踪、绘制的成本都会明显下降。这里还保留了原始时间戳，所以后面做时延统计和多传感器同步比较干净。

第三步，推理节点我没有做成单线程串行回调，而是拆成“采集/推理线程”和“后处理/发布线程”两段，中间放一个很短的队列。这个队列的策略不是追求一帧不丢，而是队列满了就丢旧帧，只保留最新帧。这样系统忙起来时，结果可能会掉一点帧，但不会出现越积越慢，也不会用旧画面去驱动机器人。这个取舍对移动机器人很关键，因为控制更看重 freshness，而不是每一帧都必须算完。

第四步，`rknn_yolov6` 会同时发布两类结果：一类是带框图像 `/camera/image_det`，一类是结构化检测框 `/ai_msg_det`。这两类消息共用同一份输入图像时间戳，所以 `object_track` 可以直接用 `ExactTime` 同步，而不需要再开一个很大的近似同步窗口。后面的跟踪节点会选出要跟踪的目标，再结合激光雷达投影到图像里估距离，最后输出控制量 `/cmd_vel`。这里的关键点不是只做视觉框选，而是把视觉识别和距离估计在同一时间基准下对齐了。

最后一层，如果要给前端或者远端看图，我会把跟踪结果再编码成 JPEG。这一步单独放在 `img_encode`，而且同样优先走板端硬件。也就是说，控制闭环用的是“缩小后的 RGB 图像 + 结构化检测结果”，显示回传走的是“压缩图像”，两条路径职责分开，不让显示链路拖慢控制链路。再加上所有节点都做了按需订阅、按需启动，整个系统在工程上会比较稳。

如果面试官让我一句话总结，我会说：这条链路的低时延核心，不是某个模型多快，而是“MJPEG 源头压缩 + V4L2 mmap + MPP/RGA/RKNN 硬件加速 + 小分辨率前置 + 短队列丢旧帧 + 时间戳贯穿全链路”。

2. 采集层：为什么入口就选 MJPEG + V4L2 mmap

核心代码在 [v4l2.h](../../src/usb_camera/src/v4l2.h#L17)、[v4l2.cpp](../../src/usb_camera/src/v4l2.cpp#L6)、[usb_camera.cpp](../../src/usb_camera/src/usb_camera.cpp#L45)。

这里有三个关键点：

- 相机格式强制为 `V4L2_PIX_FMT_MJPEG`，不是原始 YUYV。这样源头就先压了数据量。
- 驱动 buffer 用 `V4L2_MEMORY_MMAP`，减少采集路径上的额外拷贝。
- 只在有下游订阅者时才真正打开采集循环，没人看就不占资源。

关键片段：

```cpp
#define VIDEO_FORMAT V4L2_PIX_FMT_MJPEG
#define BUFFER_COUNT 4

reqbuf.memory = V4L2_MEMORY_MMAP;
reqbuf.count = BUFFER_COUNT;
ioctl(fd, VIDIOC_REQBUFS, &reqbuf);

ret = ioctl(fd, VIDIOC_DQBUF, &buf);
frame_buf->start = mmap_buffer[buf.index].start;
frame_buf->length = buf.bytesused;
ret = ioctl(fd, VIDIOC_QBUF, &buf);
```

3. 解码预处理层：为什么先解码，再立刻缩到 640x360

核心代码在 [img_decode.cpp](../../src/img_decode/src/img_decode.cpp#L7)、[mpp_decode.cpp](../../src/img_decode/src/mpp_decode.cpp#L71)、[rga_resize.cpp](../../src/img_decode/src/rga_resize.cpp#L9)。

主逻辑是：

- 订阅 `/image_raw/compressed`
- 在 ARM 板端用 MPP 做 MJPEG 硬解码
- 立刻用 RGA 做硬缩放
- 发布 `/camera/image_raw`
- 保留原时间戳

关键片段：

```cpp
int ret = mpp_decode.decode((unsigned char*)msg->data.data(), msg->data.size(), image);

msg_pub.header = msg->header;
msg_pub.height = image.rows * scale;
msg_pub.width = image.cols * scale;
msg_pub.encoding = "rgb8";
msg_pub.data.resize(msg_pub.height * msg_pub.width * 3);

rga_resize(image, msg_pub.data.data(), scale);
raw_image_pub.publish(msg_pub);
```

对应位置：

- [img_decode.cpp](../../src/img_decode/src/img_decode.cpp#L44)
- [img_decode.cpp](../../src/img_decode/src/img_decode.cpp#L72)
- [img_decode.cpp](../../src/img_decode/src/img_decode.cpp#L81)

MPP 解码有一个很有价值的实现细节：它把输出 buffer 直接包成 `cv::Mat`，而不是先 CPU `memcpy` 一份。代码里还专门写了注释，说明这块是非 cache 内存，CPU 拷贝会慢，所以后续直接交给 RGA 做缩放/拷贝更合理。

```cpp
base = (RK_U8 *)mpp_buffer_get_ptr(buffer);
// 这里的base是不带cache的内存，如果用mmcpy到用户内存会很慢，后续必须通过rga拷贝或者缩放
image = cv::Mat(height, width, CV_8UC3, base);
```

对应位置：

- [mpp_decode.cpp](../../src/img_decode/src/mpp_decode.cpp#L288)

RGA 缩放会直接把结果写到 ROS 输出消息的 `data` buffer 里，这也是一个减少中间临时对象的点：

```cpp

    STATUS = imcopy(src, dst);
```

对应位置：

- [rga_resize.cpp](../../src/img_decode/src/rga_resize.cpp#L21)

另外，这个节点也做了“按需订阅”：

- 没人订阅 `/camera/image_raw`，它就把上游 `/image_raw/compressed` 的订阅关掉
- 有人订阅时再重新订阅上游

见 [img_decode.cpp](../../src/img_decode/src/img_decode.cpp#L103)。

4. 推理层：低时延的关键不只是 RKNN，更是“短队列 + 双线程”

核心代码在 [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L26) 和 [Queue.h](../../src/rknn_yolov6/src/Queue.h#L10)。

这层真正影响时延的设计点有四个：

- 图像进入推理前，已经被前一层缩到 640x360
- 进入模型前如果尺寸不匹配，再用 RGA 做一次模型输入缩放
- 推理和后处理拆成两个线程
- 中间队列长度非常短，超了直接丢旧帧

短队列实现非常直接：

```cpp
void push(const T& value)
{
    std::unique_lock<std::mutex> lock(mutex_);
    if(queue_.size()>=2)
        queue_.pop();
    queue_.push(value);
    cond_var_.notify_one();
}
```

对应位置：

- [Queue.h](../../src/rknn_yolov6/src/Queue.h#L14)

这段代码的意义很明确：系统忙时优先“新鲜度”，不追求每帧必达。对移动机器人，这是合理取舍。

推理线程里，消息先转成 `cv::Mat`，然后放进 `capture_data_queue`，再由专门线程处理：

- 订阅回调只负责入队，见 [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L69)
- 推理线程取队列、RGA resize、`rknn_run`、`rknn_outputs_get`，见 [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L183)
- 后处理线程做 NMS、画框、发布图像和检测结果，见 [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L325)

这里还有一个很关键的同步设计：

```cpp
image_msg = cv_bridge::CvImage(infer_data.header, "rgb8", infer_data_orig_img).toImageMsg();
image_pub.publish(image_msg);

dets_msg.header = infer_data.header;
det_pub.publish(dets_msg);
```

对应位置：

- [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L454)

这意味着“画框图像”和“结构化检测结果”共用同一份 header，后面可以直接做 `ExactTime` 同步，而不是靠近似同步去撞时间窗。

一个值得学习的 caveat 也在代码注释里写出来了：

- `rknn_outputs_release` 必须及时调用
- 如果后处理长期慢于推理，而队列又在丢帧，可能会把尚未 release 的输出句柄丢掉

对应位置：

- [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L431)

这是低时延系统里很典型的工程约束：你为了 freshness 允许丢帧，但资源生命周期就必须设计得更严谨。

7. 这套低时延设计到底快在哪里

- 源头压缩：入口直接用相机 MJPEG，先压 USB 和 ROS 第一跳带宽，见 [v4l2.h](../../src/usb_camera/src/v4l2.h#L17)。
- 采集轻量：V4L2 `mmap` + 4 个 ring buffer，见 [v4l2.cpp](../../src/usb_camera/src/v4l2.cpp#L93)。
- 前置降算力：`img_decode` 先把 1280x720 缩到 640x360，再喂推理，见 [img_decode.launch](../../src/img_decode/launch/img_decode.launch#L8)。
- 硬件加速：aarch64 上编译时打开 `USE_ARM_LIB=1`，解码/缩放/编码/推理分别走 MPP、RGA、MPP、RKNN，见 [img_decode/CMakeLists.txt](../../src/img_decode/CMakeLists.txt)、[img_encode/CMakeLists.txt](../../src/img_encode/CMakeLists.txt)、[rknn_yolov6/CMakeLists.txt](../../src/rknn_yolov6/CMakeLists.txt)。
- 优先最新帧：队列长度只有 2，满了直接丢旧帧，见 [Queue.h](../../src/rknn_yolov6/src/Queue.h#L14)。
- 全链路时间戳贯穿：`usb_camera` 打戳，`img_decode` 继承 header，`rknn_yolov6` 的图像与检测结果共用 header，`object_track` 再用 `ExactTime`，见 [usb_camera.cpp](../../src/usb_camera/src/usb_camera.cpp#L156)、[img_decode.cpp](../../src/img_decode/src/img_decode.cpp#L72)、[rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L454)、[object_track.cpp](../../src/object_track/src/object_track.cpp#L483)。
- 按需启停：多个节点都根据订阅者数量决定是否继续处理上游数据，减少“没人看还一直跑”的浪费，见 [usb_camera.cpp](../../src/usb_camera/src/usb_camera.cpp#L77)、[img_decode.cpp](../../src/img_decode/src/img_decode.cpp#L103)、[rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L477)、[object_track.cpp](../../src/object_track/src/object_track.cpp#L398)、[img_encode.cpp](../../src/img_encode/src/img_encode.cpp#L72)。

8. 进一步降时延的两个“演进版”

一个是 ROS1 共享内存传输。

- 说明在 [shm_transport README](../../src/shm_transport/README.md#L8)。
- 思路不是直接用 socket 传整帧，而是把消息序列化到共享内存里，ROS 话题只传一个 handle。
- 发布端把共享内存地址转成 handle 发出去，见 [shm_publisher.hpp](../../src/shm_transport/include/shm_transport/shm_publisher.hpp#L52)。
- 订阅端拿到 handle 再去共享内存取消息，见 [shm_subscriber.hpp](../../src/shm_transport/include/shm_transport/shm_subscriber.hpp#L26)。

关键片段：

```cpp
actual_msg.data = pobj_->pshm_->get_handle_from_address(ptr);

```

```cpp

```

注意它也不是“端到端零拷贝”，因为仍然有序列化/反序列化；它解决的是“同机多进程走 loopback socket 的开销”。

这个仓库里也给了解码后的共享内存版本：

另一个是 ROS2 试验版。

- `usb_camera` 和 `img_decode` 都把 QoS 设成 `depth=1 + best_effort + volatile`，见 [ros2 usb_camera.cpp](../../ros2_ws/src/usb_camera/src/usb_camera.cpp#L66) 和 [ros2 img_decode.cpp](../../ros2_ws/src/img_decode/src/img_decode.cpp#L25)。
- `env.md` 明确写了 Fast DDS shared memory 环境准备，还把 `/dev/shm` 扩到 4G，见 [ros2 env.md](../../ros2_ws/env.md#L1)。

关键片段：

```cpp
qos_profile.durability_volatile();
```

这套配置的目标很明确：传感器链路只保留最新消息，不为可靠性去堆历史帧。

9. 代码层面的拷贝边界，你要能讲清楚

如果面试官追问“你是不是全链路零拷贝”，更准确的回答应该是：不是，但我尽量把大拷贝压到最少，而且优先让大数据在压缩态或硬件 buffer 中流动。

几个主要拷贝点是：

- 采集后发布 ROS 压缩消息时，`msg.data.assign(...)` 会把 MJPEG bytes 拷进 ROS 消息，见 [usb_camera.cpp](../../src/usb_camera/src/usb_camera.cpp#L159)。
- MPP 解码前，会先把 JPEG 数据拷进 MPP packet buffer，见 [mpp_decode.cpp](../../src/img_decode/src/mpp_decode.cpp#L173)。
- 解码输出本身没有先做 CPU 大拷贝，而是把 MPP 输出 buffer 直接包成 `cv::Mat`，见 [mpp_decode.cpp](../../src/img_decode/src/mpp_decode.cpp#L296)。
- 进入推理线程时做了一次 `clone()`，为了把消息生命周期和后续异步线程解耦，见 [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L194)。
- 后处理绘制前又 clone 了一次，避免画框污染原图 buffer，见 [rknn_run.cpp](../../src/rknn_yolov6/src/rknn_run.cpp#L331)。

所以这条链路的真实风格不是“理论最纯的零拷贝”，而是“把最贵的大数据处理尽量放在压缩态或硬件路径里，把 unavoidable copy 留在少数关键边界”。

10. 测量和验证怎么做

仓库里有专门的时延/FPS 统计节点，但没有提交固定 benchmark 数字，所以不应该凭空说“端到端就是多少毫秒”。更准确的说法是：代码已经把测量点准备好了，实际结果依赖板卡、相机、模型和当前负载。

测量节点包括：

- 普通 ROS 图像链路时延统计：[img_decode_sub.cpp](../../src/img_decode/src/img_decode_sub.cpp#L17)
- 共享内存图像链路时延统计：[shm_img_decode_test.cpp](../../src/img_decode/src/shm_img_decode_test.cpp#L16)
- 普通 ROS 压缩图像订阅统计：[test_sub.cc](../../src/usb_camera/src/test_sub.cc#L14)
- 共享内存压缩图像订阅统计：[shm_sub.cc](../../src/usb_camera/src/shm_sub.cc#L15)

这些测试都是用 `header.stamp` 到接收时刻的差值算通信时延，同时统计 FPS。

11. 你在面试里可以主动补的一句“工程判断”

这套实现的设计目标不是“每一帧必达”，而是“控制链路尽量总看最新帧”。所以我在入口用 MJPEG，前段就缩图，推理中间队列只保留短窗口，没人订阅就停上游，必要时再引入共享内存或 ROS2 depth=1 的 QoS。这一整套取舍，都是围绕实时机器人更关心 freshness 而不是 completeness 来做的。

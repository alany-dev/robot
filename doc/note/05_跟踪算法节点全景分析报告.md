## 2. 先给小白的结论

如果你只想先知道“这个节点到底在干嘛”，先看这一节。

这套系统名字里虽然叫“跟踪”，但它并不是严格意义上的“目标跟踪算法”。

更准确地说，它是：

“目标检测 + 激光雷达测距 + 简单比例控制”的机器人跟随节点。

也就是说：

1. 相机看到画面。
2. `rknn_yolov6` 检测出画面里有哪些目标，比如 `person`。
3. `object_track` 每一帧从这些检测框里，挑一个最像主目标的框。
4. 它再把激光雷达点投影到图像里，估计这个目标离机器人有多远。
5. 最后根据“目标在画面左还是右”“目标离我远还是近”，计算底盘速度 `cmd_vel`。

所以它做的是“跟人走”或者“跟目标走”的闭环控制，而不是常见论文里的那种：

- 目标重识别
- 多目标持续编号
- 跨帧身份保持
- 遮挡恢复
- Kalman 预测
- ByteTrack / SORT / DeepSORT

本项目当前实现的核心特点是：

- 视觉侧按类别筛选，默认只跟 `person`
- 在同一帧里选面积最大的检测框作为主目标
- 角速度来自图像横向偏差
- 线速度来自激光雷达距离误差
- 目标丢失时立即停下
- 只有收到 `/enable_tracking=true` 时才真正发布底盘控制速度

一句话总结：

这是一个“检测驱动的跟随控制节点”，不是“带身份保持的真正跟踪器”。

## 3. 先把三个概念分清楚：检测、跟踪、跟随控制

很多初学者会把这三个词混在一起。实际上它们不是一回事。

### 3.1 什么是目标检测

目标检测做的事情是：

- 看一张图
- 找出图里有哪些目标
- 给每个目标一个框
- 给出类别和置信度

例如：

- 框 1：`person`
- 框 2：`chair`
- 框 3：`dog`

本项目里这部分由 `rknn_yolov6` 完成。

### 3.2 什么是目标跟踪

真正的目标跟踪通常强调“跨时间连续地跟住同一个目标”。

比如连续 100 帧都在看人群，系统要尽量保证：

- 第 1 帧的 3 号人
- 第 2 帧还是 3 号人
- 第 3 帧还是 3 号人

即使暂时被挡住，或者框大小变化，它也会尝试维持同一目标身份。

常见方法有：

- 光流跟踪
- Kalman Filter
- SORT
- DeepSORT
- ByteTrack

本项目里的 `ai_msgs/Det.msg` 虽然定义了 `obj_id` 字段，但 `object_track.cpp` 并没有使用它来维持身份。

因此，本项目当前实现不能算严格意义上的“身份级跟踪”。

### 3.3 什么是跟随控制

跟随控制关心的是：

- 目标在左边还是右边
- 我要不要左转右转
- 目标离我太远还是太近
- 我要不要向前或后退

这本质上是控制问题，不是检测问题。

本项目里：

- 横向误差来自图像
- 距离误差来自激光雷达
- 输出是 `geometry_msgs/Twist`

也就是：

- `linear.x` 控制前后
- `angular.z` 控制转向

## 4. 本项目整条链路概览

### 4.1 节点级链路

这条链路可以概括成下面这样：

```text
USB 摄像头
  -> usb_camera
  -> /image_raw/compressed

/image_raw/compressed
  -> img_decode
  -> /camera/image_raw

/camera/image_raw
  -> rknn_yolov6
  -> /camera/image_det
  -> /ai_msg_det

/camera/image_det + /ai_msg_det
  -> object_track
  -> /camera/image_det_track
  -> /cmd_vel

/camera/image_det_track
  -> img_encode
  -> /camera/image_det_track/compressed

/cmd_vel
  -> base_control
  -> 电机 / 底盘运动

/scan
  -> object_track
  -> 目标距离估计

/enable_tracking, /enable_lidar
  -> object_track
  -> 开启 / 关闭跟随与激光融合
```

### 4.2 话题级总表

| 话题 | 发布者 | 订阅者 | 作用 |
|---|---|---|---|
| `/image_raw/compressed` | `usb_camera` | `img_decode` | 相机压缩图像 |
| `/camera/image_raw` | `img_decode` | `rknn_yolov6` | 解码并缩放后的 RGB 图 |
| `/camera/image_det` | `rknn_yolov6` | `object_track` | 已画检测框的图像 |
| `/ai_msg_det` | `rknn_yolov6` | `object_track` | 结构化检测结果 |
| `/scan` | `ydlidar` | `object_track` | 2D 激光雷达距离数据 |
| `/camera/image_det_track` | `object_track` | `img_encode` 或显示端 | 叠加跟踪信息后的图像 |
| `/camera/image_det_track/compressed` | `img_encode` | Web / 客户端 | 最终压缩显示图像 |
| `/cmd_vel` | `object_track` 或 Web 控制 | `base_control` | 底盘线速度和角速度 |
| `/enable_tracking` | Web / Agent | `object_track` | 开启或关闭跟随控制 |
| `/enable_lidar` | Web / Agent | `object_track` | 开启或关闭雷达融合 |

### 4.3 它为什么能低时延

这套链路设计明显偏向“最新性”，不是偏向“绝不丢帧”。

几个关键点：

- 相机先发 JPEG 压缩图，节省带宽
- `img_decode` 很早就把 `1280x720` 缩成 `640x360`
- `rknn_yolov6` 内部使用短队列，队列长度达到 2 时直接丢最旧帧
- 多个节点会根据下游有没有订阅者，动态订阅或取消订阅上游话题

这意味着它的目标不是“把每一帧都处理完”，而是“尽量让机器人拿到最新一帧结果”。

对于跟随机器人来说，这个取向是合理的，因为旧结果再准确，也比不上新结果更有控制价值。

## 5. ROS 初学者必须先懂的几个基础

### 5.1 什么是节点

ROS 里每个可执行程序通常叫一个节点。

例如：

- `usb_camera` 是一个节点
- `img_decode` 是一个节点
- `rknn_yolov6` 是一个节点
- `object_track` 是一个节点

可以把它理解成：

- 一个独立运行的程序
- 专门负责一项功能

### 5.2 什么是话题 Topic

话题就是节点之间传数据的通道。

例如：

- 相机节点把图片发到 `/image_raw/compressed`
- 解码节点订阅这个话题
- 检测节点再订阅 `/camera/image_raw`

可以把 Topic 理解成“广播频道”。

### 5.3 什么是 Header 和时间戳

很多 ROS 消息带一个 `Header`，里面最重要的是：

- `stamp`：时间戳
- `frame_id`：坐标系名称

为什么时间戳重要？

因为如果你想把“同一时刻”的图像和检测结果对应起来，就必须依赖时间戳。

本项目中：

- `usb_camera` 生成时间戳
- `img_decode` 保留原时间戳
- `rknn_yolov6` 继续沿用原时间戳
- `object_track` 因此能把图像与检测框做精确同步

### 5.4 什么是 launch 文件

`*.launch` 文件就是一组 ROS 节点和参数的启动脚本。

比如：

- `pkg_launch/launch/all.launch`

会一起拉起：

- 相机
- 解码
- 检测
- 编码
- 雷达
- 跟踪

这样不用每个节点手动开一遍。

## 6. 相机知识背景

这一节专门写给小白。很多人能看懂框，却看不懂为什么要“标定”和“投影”。

### 6.1 图像分辨率是什么

分辨率就是图像宽和高的像素数量。

本项目里：

- `usb_camera` 采集分辨率是 `1280x720`
- `img_decode` 用 `scale=0.5` 把它缩成 `640x360`

这件事非常重要，因为后面很多参数都和图像尺寸绑定。

### 6.2 像素坐标怎么理解

一张图可以看成一个二维坐标平面：

- 左上角通常是 `(0, 0)`
- 向右是 `x` 增大
- 向下是 `y` 增大

对 `640x360` 图像来说：

- 中心点大约是 `(320, 180)`

如果目标中心在：

- `x=200`，说明目标偏左
- `x=500`，说明目标偏右

于是机器人就可以根据“目标离中心偏了多少像素”来决定转向。

### 6.3 相机内参是什么

相机内参通常记成矩阵 `K`，里面最重要的是四个量：

- `fx`
- `fy`
- `cx`
- `cy`

可以粗略理解为：

- `fx`、`fy`：焦距在像素坐标里的表达
- `cx`、`cy`：主点，一般接近图像中心

它们决定了：

- 一个三维点如何映射到图像平面

本项目的标定文件 `fisheye.yml` 中保存了：

- `CameraMat`
- `DistCoeff`
- `ImageSize`

### 6.4 为什么标定文件读出来后还要“除以 2”

这是本项目非常关键、也非常容易忽略的一点。

标定文件对应的是原始图像尺寸：

- `1280x720`

但实际进入检测和跟踪链路的是缩放后的图像：

- `640x360`

如果内参还用原图尺寸对应的数值，那么投影出来的雷达点会整体偏移。

所以 `object_track.cpp` 在读取标定文件后，把：

- `fx`
- `fy`
- `cx`
- `cy`

一起缩小一半。

这就是为什么代码里有：

```text
CameraMat = CameraMat / 2
CameraMat.at<double>(2,2) = 1
```

本质上是在让相机模型和当前实际图像尺寸匹配。

### 6.5 畸变是什么

真实镜头不是完美的小孔成像模型，边缘往往会变形。

这就是畸变。

标定文件里的 `DistCoeff` 就是畸变系数。

本项目中这个文件名字叫 `fisheye.yml`，但当前 `DistCoeff` 实际全是 0：

```text
data: [0, 0, 0, 0]
```

这说明当前实现里：

- 名字虽然叫 fisheye
- 但实际投影时并没有使用真实鱼眼畸变模型
- 更像是在用普通相机模型做近似投影

这点后面会作为工程风险专门讲。

## 7. 激光雷达知识背景

### 7.1 2D 激光雷达到底测的是什么

这里的雷达是二维激光雷达，不是相机，也不是 3D 点云雷达。

它做的事情通常是：

- 在一个平面里不断旋转
- 每个角度打一束激光
- 量出离障碍物有多远

于是你会得到一圈距离值。

在 ROS 里通常就是 `sensor_msgs/LaserScan`。

### 7.2 LaserScan 里最重要的字段

`LaserScan` 里你最应该理解这几个量：

- `ranges`
- `angle_min`
- `angle_increment`
- `range_min`
- `range_max`

可以这样理解：

- `ranges[i]`：第 `i` 束激光测到的距离
- `angle_min`：第一束激光从哪个角度开始
- `angle_increment`：相邻两束之间差多少角度
- `range_min`：过近就不可信
- `range_max`：过远就不可信

本项目的 `scan_callback` 会遍历所有 `ranges`，把每一个合法测距点转成二维坐标，再变换到相机坐标系下。

### 7.3 本项目的雷达视场

`ydlidar.launch` 里配置了：

- `angle_min = 0`
- `angle_max = 180`
- `range_min = 0.1`
- `range_max = 10.0`

这说明当前更像是在使用前方半圈的测距信息。

对跟随场景来说，这通常够用，因为目标大多在机器人前方。

### 7.4 为什么光靠相机不够

相机能很好地告诉你：

- 目标在哪儿
- 看起来有多大

但它天然不擅长稳定地给出“绝对距离”。

你可以通过框大小大概猜远近，但不够稳。

激光雷达则相反：

- 不知道“这是人还是椅子”
- 但很擅长告诉你“这个方向上前方几米有东西”

所以本项目是典型的“视觉选目标，雷达估距离”。

## 8. TF 和坐标系知识背景

### 8.1 为什么必须有坐标系

相机看到的是像素坐标：

- `(u, v)`

雷达测到的是自己坐标系里的点：

- `(x, y, z)`

这两者根本不是同一种坐标。

如果不做坐标变换，你就没法知道：

- 雷达的哪个点对应图像里的哪个位置

这就是为什么 ROS 里需要 TF。

### 8.2 本项目里有哪些关键坐标系

本项目里比较关键的坐标系有：

- `base_footprint`
- `base_link`
- `camera_link`
- `laser_link`

其中：

- `camera_link` 是相机安装坐标系
- `laser_link` 是雷达安装坐标系

### 8.3 这些坐标系在机器人上大概在哪里

从 `test11.xacro` 可以看到固定安装关系：

- `camera_link` 相对 `base_link` 的位姿约为 `xyz=(0.245001, -0.006564, 0.145532)`
- `laser_link` 相对 `base_link` 的位姿约为 `xyz=(0.162817, 0.0, 0.060341)`

这说明：

- 相机更靠前、更高
- 激光雷达略靠后、较低

这也是为什么二者数据不能直接混用，必须通过 TF 变换。

### 8.4 `object_track` 怎么用 TF

`object_track` 启动时会：

1. 等待 `camera_link <- laser_link` 之间的变换可用
2. 读取这组固定外参
3. 后续把每个激光点从雷达坐标系变换到相机坐标系

这一步做完以后，雷达点才有资格被投影进图像。

## 9. 前级视觉检测节点 `rknn_yolov6`

### 9.1 它负责什么

`rknn_yolov6` 是视觉检测节点，职责包括：

- 订阅小尺寸 RGB 图像 `/camera/image_raw`
- 调用 RKNN 模型做目标检测
- 发布带检测框的图像 `/camera/image_det`
- 发布结构化检测结果 `/ai_msg_det`

### 9.2 它输出的检测消息长什么样

`ai_msgs/Det.msg` 定义了：

- `x1, y1, x2, y2`
- `conf`
- `cls_name`
- `cls_id`
- `obj_id`

`ai_msgs/Dets.msg` 则是：

- 一个 `Header`
- 一个 `Det[]`

所以可以把它理解为：

- “这一帧图像的所有检测框集合”

### 9.3 为什么 `object_track` 能精确同步图像和框

关键原因是 `rknn_yolov6` 发布两个结果时，复用了同一个输入 `header`：

- `/camera/image_det` 的 header 来自输入图像
- `/ai_msg_det` 也用同一份 header

于是 `object_track` 可以直接使用 `ExactTime` 做精确同步，而不需要模糊匹配。

这是一种很好的工程实现，因为：

- 逻辑更简单
- 同步更干净
- 避免图像和框错位

### 9.4 为什么说它也在追求低延迟

`Queue.h` 里的队列长度只有 2。

含义很明确：

- 如果新帧来得太快
- 后处理来不及
- 就扔掉最旧帧

这会带来一个很重要的系统特性：

- 机器人不会因为积压一堆旧图而“反应越来越迟”

### 9.5 一个容易忽略的细节：x86 和板端行为不同

在非 ARM 平台下，`rknn_yolov6` 不会真的跑 RKNN YOLO。

代码会退化成：

- OpenCV 灰度图
- Haar 人脸检测
- 然后把检测到的人脸标成 `person`

所以如果你在 PC 上调试，看到的是“人脸假装 person”的近似行为。

这对开发联调有帮助，但要知道它和板端真实模型行为并不一致。

## 10. `object_track` 节点总体职责

### 10.1 输入

`object_track` 的输入有三类：

- `/camera/image_det`
- `/ai_msg_det`
- `/scan`

其中：

- 前两个来自视觉检测节点
- 最后一个来自激光雷达

### 10.2 输出

它输出两类信息：

- `/camera/image_det_track`
- `/cmd_vel`

其中：

- 图像用于调试和显示
- 速度用于底盘控制

### 10.3 核心思想

它的核心思路可以用一句话概括：

“视觉负责认目标，激光负责估距离，控制器负责追上去。”

## 11. `object_track` 启动阶段做了什么

### 11.1 读取参数

`object_track.launch` 中配置了很多参数，例如：

- `tracking_class_name=person`
- `width=640`
- `height=360`
- `kp_angular=0.01`
- `kp_linear=0.5`
- `max_angular=3.0`
- `max_linear=0.15`
- `keep_distance=0.3`
- `valid_distance_min=0.05`
- `valid_distance_max=5.0`

这些参数大致决定：

- 跟谁
- 图像尺寸是多少
- 跟多快
- 保持多远
- 距离多远算有效

### 11.2 读取相机标定

节点启动后先调用 `read_camera_info_from_file()`：

- 从 `fisheye.yml` 里读 `K` 和 `D`
- 根据 `img_decode` 的缩图比例，把内参缩小到 `640x360` 对应版本

### 11.3 获取雷达到相机的固定外参

然后它会通过 TF 获取：

- `camera_frame`
- `laser_frame`

之间的固定变换。

如果这一步拿不到，后面的投影就没法做。

### 11.4 初始化发布与运行期开关

节点还会订阅两个控制话题：

- `/enable_tracking`
- `/enable_lidar`

这意味着它不是一上来就“永远工作”，而是支持运行期开关。

## 12. 图像与检测框同步：为什么这里做得比较规范

### 12.1 同步方式

`main()` 中通过 `message_filters::Synchronizer` 做：

- `sensor_msgs::Image`
- `ai_msgs::Dets`

的 `ExactTime` 同步。

这意味着只有当：

- 图像时间戳
- 检测框时间戳

完全一致时，才会触发 `sync_callback()`。

### 12.2 为什么不是 ApproximateTime

因为前级 `rknn_yolov6` 已经保证两者来自同一帧、时间戳完全一致。

这种情况下用 `ExactTime` 是合理的。

### 12.3 一个重要限制

这里同步的只有：

- 图像
- 检测框

而不是：

- 图像
- 检测框
- 激光雷达

也就是说，激光雷达 `scan_callback()` 和视觉 `sync_callback()` 不是严格时间同步的。

这会带来一个实际影响：

- 距离可能来自“最新一帧雷达”
- 框可能来自“另一时刻图像”

运动较慢时问题不大，运动较快或目标切换时就会产生误差。

## 13. 目标选择逻辑：它是怎么决定“跟谁”的

### 13.1 选择规则

在 `sync_callback()` 里，节点会遍历当前帧所有检测框，然后按下面规则选目标：

- 类别必须等于 `tracking_class_name`
- 默认就是 `person`
- 面积必须大于阈值 `object_area_threshold`
- 在满足条件的框里，选面积最大的那个

这意味着它的选择策略非常直白：

- 同类目标里，谁在画面里最大，就跟谁

### 13.2 为什么会这样设计

因为在跟人走场景里，经常可以做一个经验假设：

- 离机器人更近的人，框通常更大
- 当前主目标通常也是最近、最显眼的那个

这种策略简单、速度快、实现成本低。

### 13.3 这种策略的直接后果

优点：

- 简单
- 实时
- 不需要复杂身份管理

缺点：

- 两个人同时出现时可能换人
- 只要另一个 `person` 框更大，就可能切到新目标
- 遮挡、交叉、侧身变化时容易跳目标

所以再次强调，它不是“持续跟住某个特定身份的人”。

## 14. 控制逻辑：图像偏差怎么变成转向

### 14.1 横向误差的来源

选定目标框后，代码会取框中心：

- `object_x = (x1 + x2) / 2`

图像宽度是 `640` 时，图像中心大约是：

- `width / 2 = 320`

于是横向误差就是：

- `320 - object_x`

### 14.2 角速度公式

当前代码实际使用的是：

```text
angular.z = 0.01 * (width/2 - object_x)
```

然后再做限幅：

- 最大不超过 `max_angular`
- 最小不低于 `-max_angular`

### 14.3 给初学者一个具体例子

假设：

- 图像宽 `640`
- 图像中心 `320`
- 目标中心在 `x=200`

那么：

- 像素误差 = `320 - 200 = 120`
- 角速度 = `0.01 * 120 = 1.2 rad/s`

如果最大角速度限制是 `1.5 rad/s` 或 `3.0 rad/s`，那这个值就会被保留。

如果目标特别偏，比如在最左边，理论值可能更大，就会被限幅。

### 14.4 方向怎么理解

当目标在左边时：

- `object_x` 比 320 小
- `320 - object_x` 为正
- `angular.z` 为正

机器人就会向左转。

当目标在右边时则相反。

## 15. 控制逻辑：距离误差怎么变成前进速度

### 15.1 距离来源

距离不是从检测框大小估出来的，而是来自激光雷达投影后的最小匹配距离。

代码里对应变量是：

- `object_distance`

### 15.2 线速度公式

当前代码实际使用的是：

```text
linear.x = 0.5 * (object_distance - keep_distance)
```

只有在距离有效区间内才生效：

- `object_distance >= valid_distance_min`
- `object_distance <= valid_distance_max`

否则线速度直接给 0。

之后还会再做一次限幅：

- 最大不超过 `max_linear`
- 最小不低于 `-max_linear`

### 15.3 给初学者一个例子

假设：

- `keep_distance = 0.3 m`
- 当前估计目标距离 `0.5 m`

那么：

- 距离误差 = `0.5 - 0.3 = 0.2 m`
- 线速度 = `0.5 * 0.2 = 0.1 m/s`

也就是说机器人会向前走，直到距离接近 0.3 米。

如果目标离得太近，比如：

- `object_distance = 0.2 m`

那么：

- 线速度 = `0.5 * (0.2 - 0.3) = -0.05 m/s`

就会产生一个后退趋势。

### 15.4 `control_linear` 的作用

如果参数 `control_linear=false`，代码会强制：

- `linear.x = 0`

这时机器人只会左右转向，不会前后移动。

可以把它理解成“纯视觉对准，不做距离跟随”。

## 16. 激光融合逻辑：距离到底是怎么算出来的

这一节是整个节点里最关键、也最容易看晕的部分。

### 16.1 从 LaserScan 到三维点

对每个雷达测距点，代码做了：

1. 读取当前距离 `range`
2. 计算当前角度 `angle = angle_min + i * angle_increment`
3. 生成雷达坐标系下的点：

```text
x = range * cos(angle)
y = range * sin(angle)
z = 0
```

这一步的意思是：

- 把“某角度、某距离”的极坐标
- 变成直角坐标点

### 16.2 从雷达坐标系变到相机坐标系

之后每个点都会经过：

```text
point = laser_transform * point
```

这一步就是 TF 变换。

也就是：

- “这个点原来是站在雷达位置看见的”
- “现在换成站在相机位置来看，它在哪里”

### 16.3 手工偏移补偿

代码还支持参数：

- `offset_x`
- `offset_y`
- `offset_z`
- `offset_roll`
- `offset_pitch`
- `offset_yaw`

这说明作者考虑过实际安装误差问题。

意思是：

- 即便 URDF 里的安装位姿不是完全准确
- 也可以通过 launch 参数再做一些修正

### 16.4 从三维点投影到图像

所有转换到相机坐标系下的点，接着会被 `cv::projectPoints()` 投影到图像像素坐标。

你可以把这个过程理解成：

- 现实中的一个三维点
- 通过相机模型
- 算出它应该落在图像里的哪个像素位置

最终得到：

- `u`
- `v`

也就是：

- 图像横坐标
- 图像纵坐标

### 16.5 怎么判断雷达点属于当前目标

当前实现的筛选规则是：

1. 投影点必须落在图像范围内
2. 只要求 `u` 落在目标框的横向范围内

也就是只判断：

```text
u > x1 && u < x2
```

注意，这里没有强制要求：

```text
v > y1 && v < y2
```

### 16.6 为什么只看 x，不看 y

代码注释给出的理由是：

- 地面起伏
- 机身抖动
- 安装高度误差

可能导致纵向投影误差比较大。

所以作者想让规则更宽松一点，只用横向对齐来做关联。

### 16.7 距离最终怎么选

在所有满足条件的雷达点里，代码选的是：

- 最小距离

也就是：

- 只要某个投影点被认为“属于当前目标”
- 就拿这些点里最近的那一个做 `object_distance`

### 16.8 这个策略意味着什么

优点：

- 简单
- 保守
- 跟随时更不容易撞上目标前方的最近物体

缺点：

- 很容易受离群点影响
- 也可能把目标前方、地面、腿部附近的其他反射点当成目标距离

尤其因为当前只看 `x` 范围，不看 `y` 范围，这个风险会更明显。

## 17. `scan_callback()` 与 `sync_callback()` 的关系

### 17.1 两个回调不是一回事

`sync_callback()` 负责：

- 选目标框
- 算控制量
- 画图
- 发布 `cmd_vel`

`scan_callback()` 负责：

- 处理最新雷达扫描
- 估计当前目标距离

### 17.2 它们不是严格时间同步的

当前实现不是：

- 同一时刻图像
- 同一时刻检测框
- 同一时刻雷达

三者一起同步处理。

而是：

- 图像和检测框严格同步
- 雷达异步更新一个“最近目标距离”

所以真正控制时用到的是：

- 当前帧视觉目标
- 最近一次雷达计算出的 `object_distance`

这在静态或低速场景下通常能用，但在以下情况会变差：

- 机器人急转弯
- 目标快速横移
- 画面里多人切换
- 雷达和相机安装误差较大

## 18. 目标丢失时会发生什么

如果当前帧没有找到满足条件的目标框，代码会直接：

- `cmd_vel.linear.x = 0`
- `cmd_vel.angular.z = 0`

也就是说：

- 没目标就停

这是一个比较稳妥的默认行为。

它不会在目标丢失时继续沿用上一帧速度盲走。

## 19. 开关控制：为什么有时系统看起来“没在跑”

### 19.1 `enable_tracking`

只有当 `/enable_tracking` 收到 `true` 时，`object_track` 才会真正发布：

- `/cmd_vel`

即便前面检测都正常，如果这个开关没打开，机器人也不会动。

### 19.2 `enable_lidar`

只有当 `/enable_lidar` 收到 `true` 时，节点才会订阅：

- `/scan`

如果雷达关闭了：

- `pixel_points` 清空
- `object_distance` 清零

之后 `sync_callback()` 里由于距离不在有效区间，线速度会变成 0。

这意味着：

- 视觉还可以继续选目标
- 机器人仍可能左右转向
- 但不会根据距离向前跟进

### 19.3 整条链路是按需订阅的

这是这套系统一个很有意思的工程设计。

多个节点都在做：

- 下游没人用，我就不订阅上游
- 下游有人用了，我再恢复上游订阅

结果就是整条链路会出现“按需点亮”的效果。

例如：

1. 如果没有人订阅最终跟踪图，也没有启用跟随
2. `object_track` 就会取消对上游图像和检测框的订阅
3. `rknn_yolov6` 发现没人订阅检测图，也会停止订阅原始图
4. `img_decode` 发现没人订阅解码图，也会停止订阅相机压缩流
5. `usb_camera` 没有订阅者时也不会继续发布

所以你有时会感觉：

- “怎么节点都在，但图像不流动？”

这不是坏了，而是它在省资源。

## 20. 最关键的实现伪代码

### 20.1 `sync_callback()` 逻辑伪代码

```text
输入：一帧 image_det + 同时间戳 dets

1. 遍历 dets
2. 只保留 cls_name == tracking_class_name 的框
3. 选面积最大的框作为 object_det
4. 如果找到目标：
   - object_x = 框中心 x
   - angular.z = 0.01 * (图像中心 - object_x)
   - 如果 object_distance 在合法范围：
       linear.x = 0.5 * (object_distance - keep_distance)
     否则：
       linear.x = 0
   - 对 linear / angular 做限幅
   - 画框、画文本
5. 如果没找到目标：
   - linear.x = 0
   - angular.z = 0
6. 如果 enable_tracking=true：
   - 发布 /cmd_vel
7. 把投影雷达点画到图上
8. 发布 /camera/image_det_track
```

### 20.2 `scan_callback()` 逻辑伪代码

```text
输入：一帧 LaserScan

1. 遍历所有 ranges
2. 跳过 NaN、过近、过远点
3. 用 range + angle 算出雷达坐标系点
4. 用 TF 变换到相机坐标系
5. 加上人工偏移补偿
6. 调用 projectPoints 投影到图像
7. 如果像素点落在图像内：
   - 保存到 pixel_points 用于调试显示
8. 如果像素 u 落在当前目标框的 x 范围内：
   - 计算三维距离
   - 取最小值作为 object_distance
9. 更新 pixel_points 和 object_distance
```

## 21. 这套实现的优点

### 21.1 优点一：结构非常清晰

它把任务拆成了几个简单模块：

- 相机采集
- 解码缩放
- 视觉检测
- 雷达测距
- 控制输出

这比把所有逻辑揉成一个大节点更容易理解和维护。

### 21.2 优点二：对嵌入式比较友好

这套实现有明显的嵌入式思路：

- 尽早缩图
- 短队列
- 动态订阅
- 用检测驱动，而不是复杂多目标跟踪

这些都非常符合板端资源有限的现实。

### 21.3 优点三：调试友好

`object_track` 会把这些东西直接画在图上：

- 目标框
- 投影雷达点
- 目标像素位置
- 目标距离
- 当前控制速度

这对于定位问题非常有帮助。

### 21.4 优点四：控制逻辑简单直接

这套控制不是黑箱。

它就是：

- 像素偏差 -> 转向
- 距离偏差 -> 前进

非常适合教学、演示和小型服务机器人场景。

## 22. 这套实现的缺点、风险和局限

这一节很重要。前面讲的是“它怎么工作”，这里讲“它哪里不够好”。

### 22.1 它不是真正的身份跟踪

最核心的问题是：

- 没有持续身份管理
- 没有跨帧追踪
- 没有预测器
- 没有丢失恢复

直接后果就是：

- 只要场景里出现另一个更大的 `person`
- 目标就可能切换

### 22.2 `kp_angular` 和 `kp_linear` 参数读了，但没真正用

这是当前代码里一个非常明确的工程问题。

launch 文件和参数区里确实定义了：

- `kp_angular`
- `kp_linear`

但是在实际控制计算时，代码写死的是：

- `0.01`
- `0.5`

而不是使用读进来的参数变量。

这意味着：

- 你在 launch 里改 `kp_angular`
- 控制可能并不会跟着变

这属于“参数与实际实现不一致”的典型问题。

### 22.3 雷达和视觉没有做真正时间同步

这会导致：

- 框是这一刻的
- 距离可能是另一刻的

低速时能凑合用，高速时误差会放大。

### 22.4 雷达点关联只看 x 范围，不看 y 范围

这虽然是为了增强鲁棒性，但副作用很明显：

- 目标框同一列上的其他物体
- 地面杂点
- 腿部附近的误投影

都可能被算进来。

### 22.5 距离取最小值，容易受离群点影响

只要某个误匹配的点更近，它就会主导 `object_distance`。

这可能让机器人误以为：

- “目标已经很近了”
- “前面突然有东西”

从而导致不该停的时候停，不该减速的时候减速。

### 22.6 标定模型和文件命名不完全一致

文件名叫 `fisheye.yml`，但：

- 当前用的是普通 `cv::projectPoints`
- 畸变参数还是 0

这说明当前标定和模型选择未必完全严格匹配真实镜头。

如果镜头视场较大，边缘投影误差可能会明显。

### 22.7 偏移旋转可能存在重复应用风险

代码里一方面：

- 先对点做了欧拉角旋转

另一方面：

- 又把 `offset_roll/pitch/yaw` 写进 `projectPoints` 的 `rvec`

如果这些偏移参数不是 0，就有可能出现“旋转补偿被算了两次”的风险。

目前 launch 里默认都是 0，所以平时不一定暴露，但设计上值得注意。

### 22.8 某些异常路径下旧距离可能短时残留

例如在 `scan_callback()` 里：

- 如果 `camera_points.empty()`
- 或某些投影路径提前 `return`

就不会更新新的 `object_distance`

这意味着理论上可能短时间沿用上一帧距离。

这不是高概率大问题，但从鲁棒性上说，最好在异常路径里显式清空旧值。

### 22.9 直接改写输入图像消息 buffer，不是最稳妥的 ROS 写法

当前 `sync_callback()` 中直接在输入消息的像素 buffer 上作图，然后再发布。

这种方式能省一次拷贝，但工程上有两个风险：

- 改写了原始输入消息
- 代码可读性和安全性稍差

更稳妥的做法通常是：

- 新建输出消息
- 明确把处理结果作为新消息发布

### 22.10 x86 调试结果和真实板端不一致

在 PC 上调试时看到的“person”可能只是人脸检测结果。

所以：

- 能验证话题链路
- 能验证跟随逻辑
- 但不能完全代表板端 YOLO 检测效果

## 23. 为什么有时机器人只会转，不会前进

这也是初学者最常见的问题之一。

出现“只转不前进”通常有下面几种原因：

### 23.1 雷达没有启用

如果 `/enable_lidar=false`：

- `object_distance` 会被清成 0
- 距离不在有效区间
- 线速度变成 0

### 23.2 没有匹配到有效雷达点

比如：

- 投影点没落到目标横向范围里
- 标定不准
- TF 不准
- 目标太近或太远
- 当前雷达看不到目标

这样 `object_distance` 就不会给出一个合法值。

### 23.3 距离超出有效范围

当前代码要求距离在：

- `0.05 m ~ 5.0 m`

之间才允许前后速度。

如果距离超出这个范围，线速度就是 0。

### 23.4 `control_linear` 被关掉

如果这个参数是 `false`，代码会强制把线速度清零。

## 24. 为什么目标会突然换人

这也是这个系统最典型的现象之一。

根本原因很简单：

- 每一帧都重新选
- 规则是“同类里谁面积最大就跟谁”

因此只要发生以下情况，就可能切人：

- 另一个人更靠近机器人
- 原目标暂时被遮挡
- 两个人交叉通过
- 框抖动导致大小关系反转

这不是 Bug，而是当前算法定义本身决定的行为。

## 25. 系统启动与运行时序

### 25.1 总启动入口

总体上，ROS 视觉跟随链路由：

- `pkg_launch/launch/all.launch`

启动：

- `usb_camera`
- `img_decode`
- `rknn_yolov6`
- `img_encode`
- `ydlidar`
- `object_track`

### 25.2 底盘控制为什么不在 `all.launch`

`all.launch` 中把：

- `base_control.launch`

注释掉了。

但在 `src/config/start.sh` 里，系统会单独启动：

- `roslaunch base_control base_control.launch`

所以底盘控制并不是没启动，而是放到了系统启动脚本里单独拉起。

### 25.3 跟随模式的外部控制入口

Web 控制和 Agent 控制都会发布：

- `/enable_tracking`
- `/enable_lidar`

常见控制入口包括：

- `web_controller.py`
- `src/agent/robot_following_controller.py`

这意味着“开始跟随”并不是 `object_track` 自己决定的，而是外部系统发命令给它。

## 26. 初学者调试指南

下面这一节尽量写成“照着做就能看懂”的形式。

### 26.1 第一步：确认节点是否都起来了

常用命令：

```bash
rosnode list
```

你应该重点关注这些节点是否存在：

- `/usb_camera`
- `/img_decode`
- `/rknn_yolov6`
- `/object_track`
- `/img_encode`
- `/ydlidar_node`
- `/base_control`

### 26.2 第二步：确认话题是否存在

```bash
rostopic list
```

重点看：

- `/image_raw/compressed`
- `/camera/image_raw`
- `/camera/image_det`
- `/ai_msg_det`
- `/camera/image_det_track`
- `/camera/image_det_track/compressed`
- `/scan`
- `/cmd_vel`

### 26.3 第三步：确认图像链路是否通

推荐按顺序看：

1. 相机原始压缩图
2. 解码后图
3. 检测后图
4. 跟踪叠加图

如果你装了图像查看工具，可以看：

- `/camera/image_det`
- `/camera/image_det_track`

如果最终显示链路走压缩图，则看：

- `/camera/image_det_track/compressed`

### 26.4 第四步：确认检测框有没有出来

```bash
rostopic echo /ai_msg_det
```

你应该能看到：

- `header`
- `dets`
- 每个框的 `x1 y1 x2 y2 cls_name conf`

如果一直是空：

- 可能模型没跑起来
- 可能当前场景没有检测到 `person`
- 也可能 x86 环境下 Haar 检测没检测到人脸

### 26.5 第五步：确认雷达有没有数据

```bash
rostopic echo /scan
```

你应该看到：

- `ranges`
- `angle_min`
- `angle_increment`

如果这里没数据，`object_track` 就算看到了目标，也拿不到稳定距离。

### 26.6 第六步：确认开关有没有打开

```bash
rostopic echo /enable_tracking
rostopic echo /enable_lidar
```

如果没有被设置成 `true`：

- 机器人不会真正进入完整跟随状态

### 26.7 第七步：确认底盘有没有收到速度

```bash
rostopic echo /cmd_vel
```

如果目标居中且距离合适，可能会接近：

- `linear.x = 0`
- `angular.z = 0`

如果目标在左边，应该出现正的角速度。

如果目标在前方但偏远，应该出现正的线速度。

### 26.8 第八步：看 TF 是否正常

如果相机和雷达投影明显错位，要重点怀疑：

- `camera_link`
- `laser_link`
- URDF 安装位姿
- 标定文件

这一类问题通常表现为：

- 图像里目标明明在中间
- 但雷达距离总是匹配不上
- 或投影点明显飘到错误位置

### 26.9 出问题时优先排查顺序

推荐按这个顺序查：

1. 目标检测是否正常
2. `/enable_tracking` 是否打开
3. `/enable_lidar` 是否打开
4. `/scan` 是否正常
5. TF 是否正常
6. 标定和缩放是否匹配
7. `/cmd_vel` 是否已经算出来
8. `base_control` 是否收到并执行

## 27. 给初学者的“现象 -> 原因”速查表

### 27.1 现象：图上有框，但车不动

可能原因：

- `enable_tracking` 没开
- `base_control` 没启动
- `/cmd_vel` 没被发布

### 27.2 现象：车会左右转，但不往前走

可能原因：

- `enable_lidar` 没开
- 没匹配到有效雷达距离
- 距离超出有效区间
- `control_linear=false`

### 27.3 现象：目标总是切换

根本原因：

- 当前不是身份跟踪
- 只是每帧选择最大的 `person`

### 27.4 现象：雷达点投影位置明显不对

重点怀疑：

- 标定文件
- 缩放后的内参
- TF 外参
- 偏移补偿参数

### 27.5 现象：PC 上看起来能用，上板后行为不一样

可能原因：

- x86 是 Haar 人脸近似
- 板端才是真正的 RKNN YOLO

## 28. 后续改进建议

如果后面要把这套系统做得更稳、更像真正的“跟踪系统”，建议按下面顺序改。

### 28.1 第一优先级：把参数真正用起来

先修正：

- `kp_angular`
- `kp_linear`

让代码真正使用 launch 中配置的参数值，而不是硬编码常数。

这是投入最小、收益最直接的改进。

### 28.2 第二优先级：补真正的时间同步

更理想的做法是把：

- 图像
- 检测框
- 激光

做更严格的时间对齐，至少要降低异步误差。

### 28.3 第三优先级：改进雷达点关联策略

可以考虑：

- 同时约束 `x` 和 `y`
- 只取框下半部分
- 做距离中值滤波
- 做点数阈值判断
- 用聚类后再选最近簇

这会明显提高距离估计稳定性。

### 28.4 第四优先级：加入真正的目标跟踪器

如果希望“跟住同一个人”，建议引入：

- SORT
- ByteTrack
- DeepSORT

至少要做到：

- 不再每帧只按面积选目标
- 而是根据历史轨迹保持目标身份

### 28.5 第五优先级：重新做相机标定

建议确认：

- 镜头到底是不是鱼眼
- 应该使用普通相机模型还是鱼眼模型
- 畸变参数是否真的为 0

如果标定不准，雷达投影再复杂也不会稳定。

### 28.6 第六优先级：完善异常路径

建议在这些情况显式清空状态：

- 没有有效雷达点
- 投影失败
- 当前没有目标框

避免旧的 `object_distance`、`pixel_points` 被短时间沿用。

## 29. 最终结论

本项目的“跟踪算法节点”本质上不是经典意义上的多目标跟踪器，而是一套面向机器人跟随场景的工程化融合控制节点。

它的核心逻辑非常清楚：

- 相机负责看见目标
- 检测网络负责找出目标框
- `object_track` 负责在候选框里挑出要跟的人
- 激光雷达负责补距离
- 控制器把像素误差和距离误差变成 `cmd_vel`
- `base_control` 再把 `cmd_vel` 落到底盘电机

它的优点是：

- 简单
- 实时
- 嵌入式友好
- 非常适合教学、演示和中低复杂度跟随场景

它的局限也很明确：

- 不是身份级跟踪
- 容易换目标
- 距离融合较粗糙
- 雷达与视觉不同步
- 参数和实现存在不一致

如果你是初学者，最重要的认知不是去背每一行代码，而是先建立下面这个整体图景：

```text
相机给“方向”
雷达给“距离”
检测给“目标是谁”
控制器给“怎么动”
底盘把“怎么动”变成真实运动
```

当你把这个闭环想清楚之后，再回头看 `object_track.cpp`，很多代码就不再是“黑魔法”，而是一个个可以解释的工程步骤。

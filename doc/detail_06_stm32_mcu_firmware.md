# 下位机 STM32 MCU 固件：深度版学习结果

## 1. 30 分钟路线的实际收获：先看清 MCU 真正负责什么

我替你把 `MCU/user/main.c`、`encoder.c`、`pwm.c`、`usart3.c` 串起来之后，可以先给你一个很重要的结论：MCU 不是“小电脑版上位机”，它的职责被刻意压缩成实时执行层。

它真正负责的是：

- 每 20ms 固定读一次编码器。
- 把最近收到的 PWM 命令作用到电机。
- 如果命令超时就强制停车。
- 把编码器数据打包回传给上位机。

也就是说，底层追求的是确定性和安全性，而不是功能丰富。

## 2. 2 小时路线的实际收获：控制周期里到底发生了什么

### 2.1 `main.c` 里最重要的是 `SysTick_Handler`

`SysTick_Handler()` 就是底层控制主循环，而且频率是 50Hz，因为：

- `CYCLE_TIMER_S = 0.02`
- 也就是每 20ms 触发一次

这个中断里依次做了：

1. `read_encoder_tim2()` 读取左轮脉冲。
2. `read_encoder_tim4()` 读取右轮脉冲，代码里取了负号以统一方向定义。
3. 如果收到新命令 `new_recv_flag==1`，就刷新 `pwm1/pwm2`，并把心跳计数清零。
4. 如果 5 个周期都没收到新命令，`heart_beat_cnt >= 5`，就把两个 PWM 强制清零。
5. `pwm_ouput_tb6612(pwm1, pwm2)` 输出到电机驱动。
6. 把编码器值写回 `mcu_data` 并通过串口逐字节发送。

如果你是新手，这里最值得学的是：底层循环非常短，不在中断里做复杂业务，只做“读反馈、执行命令、做保护、发状态”。

### 2.2 为什么心跳超时是 100ms

代码里是：

- 控制周期 20ms
- 连续 5 次没有新命令就停车

也就是 100ms 左右。这个时间很有工程意味：

- 太短会让通信稍一抖动就误停车。
- 太长会让上位机死掉后底盘还继续滑行太久。

100ms 是一个比较实用的折中。

### 2.3 编码器为什么用 TIM2 / TIM4 硬件编码器模式

`encoder.c` 里不是普通 GPIO 中断计数，而是：

- 定时器配置成 `TIM_EncoderInterfaceConfig(..., TIM_EncoderMode_TI12, ...)`
- 每次读取 `TIMx->CNT`
- 读完马上清零

好处是：

- CPU 不用自己逐个边沿数脉冲。
- 高速转动时不容易漏计。
- 每个 20ms 周期直接拿到“这一周期的增量脉冲”，非常适合速度闭环。

### 2.4 `pwm.c` 里真正做的是“方向 + 占空比”两件事

`pwm_ouput_tb6612()` 不是简单给一个 PWM 数：

- 若 `pwm1 < 0`，先把它变正，再切换左轮方向 GPIO。
- 若 `pwm2 < 0`，同理切换右轮方向 GPIO。
- 最后把绝对值分别写入 `TIM1` 的比较寄存器。

这说明“负 PWM”不是 MCU 直接输出负电压，而是通过方向引脚 + 正 PWM 占空比共同实现正反转。新手经常在这里概念混淆。

### 2.5 `usart3.c` 里串口协议为什么可靠一些

串口中断里会不断把收到的字节放进 `rx_data`，并通过最近 4 个字节判断是否看到了：

- `T`
- `A`
- `\r`
- `\n`

一旦完整尾部到齐，就调用 `receive_finished_process()`，再检查：

- 帧头是不是 `DA`
- 长度字段 `rx_data[2]` 是否等于实际 `rx_cnt`
- 实际长度是否等于 `sizeof(CmdData)`

全部通过后，才 `memcpy` 到 `cmd_data`，并设置 `new_recv_flag=1`。

这说明 MCU 没有用“收几个字节就信了”的脆弱策略，而是做了最基本的数据帧校验。

### 2.6 代码里还有一个很真实的小细节：上电自检式动作

`main()` 里在正式启动 SysTick 前，曾经调用：

- `pwm_ouput_tb6612(1200,1200)`
- `delay_nop_ms(500)`
- `pwm_ouput_tb6612(0,0)`

这看起来像一个简单的上电测试动作。它不一定适合最终量产版，但能反映开发阶段常见做法：上电先轻推一下电机，确认驱动链路正常。

## 3. 1 天路线的实际收获：如果你亲手验证，应当看到什么

如果你花一天验证这部分，最值得观察的是：

1. 正反转时编码器计数符号是否和上位机假设一致。
2. 20ms 周期里回传帧是否稳定，没有丢帧或粘包异常。
3. 拔掉上位机或停止发命令后，约 100ms 是否触发停车。
4. 左右轮方向引脚与 PWM 输出是否和期望一致。
5. 如果上位机里程计方向反了，问题是编码器符号、轮子安装，还是上位机约定不同。

## 4. 原文问题逐一回答

### 4.1 为什么 MCU 更适合做固定频率实时执行，而不是复杂业务编排

因为 MCU 最擅长的是时序确定、外设控制和低延迟中断响应，不擅长跑复杂依赖、网络服务和高层逻辑。把底盘闭环和安全保护放在 MCU，可以让控制节拍更稳定；把复杂业务放在上位机，则更易迭代和维护。

### 4.2 编码器接口为什么优先用硬件定时器模式

因为编码器脉冲可能很快，若靠软件轮询或普通中断统计，CPU 更忙、漏计风险更高。定时器硬件本来就是干这个的，既更准，也更省资源。

### 4.3 串口协议为什么必须设计成带帧结构

因为串口只是字节流，没有天然消息边界。没有帧头、长度、帧尾，你很难知道当前读到的是不是一条完整命令，也很难从错位或脏数据里恢复。带帧结构能显著提高鲁棒性。

### 4.4 100 ms 心跳超时保护解决了什么风险

它解决的是“上位机死了，但机器人还沿着上一次命令继续跑”的风险。对于真实机器人，这比单纯控制误差更危险，因为会演变成安全事故。

### 4.5 上位机和下位机的职责边界为什么要划得这么清楚

因为边界一旦模糊，调试会非常痛苦。你会分不清一个问题是运动学公式错了、PID 错了、串口错了，还是 MCU 驱动错了。当前结构把“高层目标”和“实时执行”分开后，问题定位更清晰，系统也更稳定。

## 5. 给小白的补充背景

- SysTick 是 MCU 里的系统滴答定时器，常用来做固定周期任务。
- TB6612 是常见双路直流电机驱动芯片，方向和 PWM 是分开的。
- “脉冲增量”本质上就是这一小段时间里轮子转过了多少格。
- 固件代码看起来不花哨，但它决定了底盘是否安全可靠，这往往比花哨更重要。

# ROS1 / ROS2 通信切换与时延性能分析：深度版学习结果

## 1. 30 分钟路线的实际收获：先看出“迁移了什么，没迁移什么”

我替你对照读完 `src/usb_camera/src/usb_camera.cpp`、`src/img_decode/src/img_decode.cpp` 和 ROS2 工作区里的同名文件后，可以先下一个结论：作者迁移的主要是通信框架和节点 API，没有推翻业务链路。

例如相机节点里，ROS1 和 ROS2 版本都保留了这些核心策略：

- 只有存在订阅者时才真正打开相机设备。
- 相机打不开时轮询 `/dev/video0~2` 做设备切换。
- 图像 header 时间戳在采集时就打上。
- 30 帧统计一次 FPS。

这说明作者不是在比较“两种完全不同写法谁更快”，而是在比较“同一条业务链路迁到 ROS2 后，怎样继续做低时延”。这点非常重要，因为公平对比的前提就是业务逻辑尽量一致。

## 2. 2 小时路线的实际收获：真正的差异在 QoS 和 DDS 传输层

ROS2 相机节点和解码节点最值得你记住的不是 `rclcpp` 语法，而是这组 QoS：

- `rclcpp::QoS(1)`
- `best_effort()`
- `durability_volatile()`

翻成大白话就是：

- 队列深度只保留最新 1 帧。
- 可以容忍丢包，不强求每帧必达。
- 不保存历史消息，后来者不用追旧账。

这是一种非常明确的实时性策略。对于图像流来说，最怕的不是偶尔丢一帧，而是排队以后处理一堆过期帧。作者这里是在主动用“可能少量丢帧”换“延迟不堆积”。

`ros2_ws/src/img_decode/src/img_decode.cpp` 也延续了 ROS1 的懒订阅思路，不过实现从“独立线程检查”换成了 `create_wall_timer(100ms)`。逻辑仍然是：

- 下游没人订阅，就 `compressed_image_sub.reset()`。
- 一旦有人订阅，再创建压缩图订阅。

所以你要得出的结论不是“ROS2 更先进”，而是“作者把低时延设计原则迁到了 ROS2”。

## 3. 1 天路线的实际收获：怎样真正做性能分析

`ros2_ws/src/usb_camera/src/test_sub.cc` 提供了最小测量点：

- 记录收到第一帧时间作为起点。
- 计算相邻两帧间隔得到瞬时 FPS。
- 读取消息 header stamp，与接收时刻相减得到单帧延迟。
- 每 30 帧打印平均 FPS、最大/最小 FPS 和当前消息延迟。

这只是测量起点，不是最终答案。真正要做对比，你至少要控制这些变量：

- 输入源一致：同一相机、同一分辨率、同一压缩格式。
- 运行环境一致：同一台机器、相近 CPU 负载。
- 节点职责一致：都走“采集 -> 解码 -> 订阅测量”路径。
- 优化开关清楚：ROS2 默认 DDS vs Fast DDS SHM 配置要分开测。

`ros2_ws/env.md` 和 `shm_fastdds.xml` 说明作者还把 DDS 层往共享内存方向继续推了一步：

- `RMW_IMPLEMENTATION=rmw_fastrtps_cpp`
- `FASTRTPS_DEFAULT_PROFILES_FILE=.../shm_fastdds.xml`
- XML 中 `useBuiltinTransports=false`，显式禁用默认 UDP/TCP。
- `userTransports` 只启用 SHM。

这意味着 ROS2 不只是换了 API，还尝试把中间件数据搬运路径也压缩了。

## 4. 关键实现细节，我替你翻成白话

### 4.1 ROS1 相机节点已经是懒加载

很多人会以为“只有 ROS2 才有更先进机制”。其实 ROS1 版 `usb_camera.cpp` 就已经做到：当 `pub.getNumSubscribers()==0` 或 `enable_camera==0` 时，不采集图像；一旦没人订阅，直接释放 `v4l2` 设备。ROS2 只是把同样思想迁过去。

### 4.2 ROS2 里的关键不是语法，而是默认策略被改得更激进

ROS2 相机和解码节点都把发布者 QoS 设成深度 1，这一点比把回调从 `ros::spinOnce()` 改成 `rclcpp::spin_some()` 更重要。因为这直接决定消息是否积压。

### 4.3 Fast DDS SHM 配置为什么重要

很多人学 ROS2 只学到 QoS，就停了。但这里作者连 DDS 传输层都管了。`segment_size`、`port_queue_capacity`、`healthy_check_timeout_ms` 都在影响共享内存传输的稳定性和等待时间。也就是说，时延优化不是只改节点代码，还改了中间件行为。

## 5. 原文问题逐一回答

### 5.1 这套仓库为什么没有简单地“只保留 ROS2”

因为这个项目要解决的是“低时延机器人系统怎么做”，不是“API 迁移打卡”。保留 ROS1 和 ROS2 两套实现有两个价值：

- 方便在已有 ROS1 生态上继续跑整机。
- 方便做结构相近的延迟对照实验，判断 ROS2 到底值不值得迁。

如果直接删掉 ROS1，你反而失去了参照物。

### 5.2 ROS2 里哪些 QoS 选择是在主动换取低延迟

核心就是三项：深度 1、`best_effort`、`volatile`。这三项一起决定了系统偏好“只保留最新数据，不背历史包袱”。它的代价是可能丢数据，但收益是减少排队和堆积。

### 5.3 Fast DDS 共享内存配置为什么会显著影响结果

因为 ROS2 节点层写得再轻，如果 DDS 还在走默认网络传输，数据依然可能绕更长路径。XML 里直接关闭 builtin transports，强制用 SHM，本质上是在减少中间件层的数据搬运成本。这对高频大消息尤其明显。

### 5.4 如何设计一组相对公平的 ROS1 / ROS2 时延对比实验

你应该固定：输入设备、分辨率、压缩格式、CPU 负载、节点拓扑、测量点。然后至少测三组：

- ROS1 原始链路。
- ROS2 普通链路。
- ROS2 + Fast DDS SHM 优化链路。

输出不能只看平均延迟，还要看 P95、帧率波动和长时间运行是否积压。

### 5.5 为什么 README 里的“性能分析”需要你自己补完实验数据，而不是只看现有代码

因为仓库里给的是实现和测量抓手，不是最终实验报告。代码能告诉你“作者打算怎么优化、怎么测”，但不能替你回答“在你的机器、你的输入、你的负载下，最终是多少毫秒”。性能问题天然依赖环境，必须自己复现。

## 6. 给小白的补充背景

- ROS1 和 ROS2 最大差别之一，不只是 API，而是 ROS2 底层用了 DDS，中间件可调空间更大。
- QoS 可以理解成“消息投递规则”。深度、可靠性、是否保留历史，都会影响延迟和稳定性。
- 对摄像头这种连续流数据，实时性系统往往更看重“新鲜”而不是“完整”。
- 做性能分析时，最怕的是比较对象不公平。代码结构差很多时，结论就不可信。
